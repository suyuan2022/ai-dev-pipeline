6 / 28

最小必要干预

模型本身是有能力的。你要给它发挥空间，如果全部定死了反而不行。
关键是：最少的干预，最大的效果。

经验预置

AI 第一次用工具，可能要试十几遍才摸对路。这种经验可以提前帮它备好，让它下次一步到位。

例子：TDD 的红绿重构循环、代码审查的 8 条铁律、LSP 的精准定位 — 都是把"老手的经验"预置给 Agent，引导它往正确方向走，从源头反制 AI slop。

循环机制

AI 聊着聊着就会"漂移"— 忘了自己原本的任务是什么。加一个循环（loop），拉回来，但不限制它怎么想。

例子：/grill 里有一个循环 — "沿着决策树每个分支走下去，逐一解决，直到所有分支穷尽"。这保证 Agent 不会聊到一半就收场，又不限制它怎么提问。

既给 Agent 发挥空间，又确保它不脱离目标。管好 Agent ≠ 管死 Agent。

/grill — 拷问你的想法 你参与

AI 扮演严格的产品经理，不厌其烦地追问。把你脑子里知道但 Agent 不知道的东西全部说出来。

Skill 核心指令（中文翻译）
不厌其烦地拷问我关于这个方案的每一个方面，直到我们达成共识。沿着设计树的每一个分支走下去，逐一解决决策之间的依赖关系。每个问题给出你推荐的答案。

一次只问一个问题，等我回答之后再问下一个。

如果一个问题可以通过探索代码库来回答，那就自己去查代码，不要问我。

当用户使用的术语跟 CONTEXT.md 里的定义冲突时，立刻指出。当用户用模糊的词，提出精确的规范用语。用具体场景来压力测试设计。

把想法变成需求文档

Skill 核心指令（中文翻译）
把当前对话的上下文和代码库理解合成为 PRD（需求文档）。不要再次拷问用户 — 直接综合已有信息。只做确认性追问。

PRD 质量规则：下一个 Agent 永远无法访问本次对话。必须列出每一个决策、约束、边界条件、被否决的替代方案 — 不允许省略。缺一项就是缺陷。

PRD 写入 Issue（一种任务工单） → 全新 AI 会话直接读取 → 带着完整上下文开干。

为什么要这么严格？下一个 Agent 看不到你和 /grill 的对话。PRD 是唯一的信息传递渠道。漏一个决策，Agent 就会自己猜——猜错就是返工。

LSP — 精准定位，省下大量上下文

LSP（语言服务协议）是代码编辑器的标配能力。配好后 Agent 可以精准定位代码，而不是把整个项目读一遍。

类比：代码字典

"这个函数在哪定义？" → 一步跳转。
"谁在调用它？" → 列出所有引用。

没有 LSP，Agent 靠全局搜索猜测，经常猜错还浪费大量上下文。有了它，定位精准、上下文占用极少。

设置方式：通过 /lsp-setup Skill 一键配置，支持 TypeScript、Python、Go 等主流语言。

对比：有 vs 没有

没有 LSP

"改 handlePayment 函数" → Agent 搜索 15 个文件、读 3000 行代码来找这个函数 → 上下文消耗大量 token

有 LSP

"改 handlePayment 函数" → 跳转到定义（1 行命令）→ 查引用（列出 4 个调用方）→ 精准修改 → 上下文几乎没浪费

让 Agent 待在聪明区

三个手段，控制上下文消耗。

精准定位

有了 LSP，Agent 不需要把代码库读一遍。一步跳转到定义，查引用列出所有调用方，精准修改。省下的上下文留给真正重要的思考。

Skill 聚焦

每个 Skill 只给 Agent 必要的上下文。/grill 只管需求对齐，/tdd 只管写代码，/bugfix 只管修 Bug。不让无关信息占位置。

会话接力 /handoff

上下文快满了 → 压缩为交接文档 → 新会话从断点继续。Bug 调试带 Debug Chain（调试链），不走弯路。

/handoff Skill 核心指令（中文翻译）
写一份交接文档，让全新的会话能接续工作。自动获取当前会话 ID，压缩对话为交接文档，引用已有产物而不是复制。Bug 场景包含 Debug Chain：当前阶段 + 反馈环 + 已排除的假设 + 关键发现。

知识沉淀 — Agent 越用越懂你的项目

CONTEXT.md

业务术语表

"用户每日可调用 100 次 API" — 写进去后 Agent 再也不会问，更不会自己改成 50。

docs/adr/

技术决策记录

"为什么选 SQLite 而不是 PostgreSQL？" — Agent 不知道历史，可能做相反决策。ADR 记录选择和原因。

docs/specs/

已知陷阱

"金额计算不能用浮点数" — 修这块代码前自动读取，不再踩坑。

知识三件套越积累越厚。Agent 每次新会话先读 → 不重复问已确认的事 → 不重复犯已踩过的坑 → 产出质量越来越高。这就是"让 Agent 越来越聪明"的长期手段。

/tdd — 先写测试再写代码

红
先写一个测试（会失败，因为代码没写）
绿
写最少的代码让测试通过
重
优化代码结构，测试仍然通过
Skill 核心指令（中文翻译）
核心原则：测试应该通过公共接口验证行为，而不是验证实现细节。代码可以完全重写，测试不该受影响。

好的测试是集成式的：通过公共 API 走真实代码路径。描述的是系统"做什么"而不是"怎么做"。

规则：一次一个测试，只写刚好够通过的代码。红灯时绝不重构。先变绿再优化。
/tdd 执行流程示例
AI
读取需求文档，确认需要实现 3 个行为：创建邀请码、验证邀请码、过期处理。
AI
切片 1/3 — 红：写测试 "创建邀请码返回 8 位字符串，状态为 active"
AI
测试失败（预期）。绿：实现 createInviteCode()，测试通过。
AI
切片 2/3 — 红：写测试 "使用已用过的邀请码应返回错误"
AI
绿：实现验证逻辑。重构：提取 validateInviteCode()。全部测试仍通过。

为什么有用？ Agent 不能自己判断做对没有。测试是客观裁判——过了就是过了，没过就是没过。方法论逼它用闭环方式完成任务。

代码审查 — Thermo-Nuclear Review

来自 Cursor 团队的代码审查方法论。核心理念：Code Judo（代码柔术） — 用更好的结构让代码消失，而不是整理代码。

不要满足于"代码能跑"。要追问：有没有一种重构方式，能让整段分支、整个 helper、整层抽象直接消失？
0

大胆追求结构简化

不要停在"这里可以更干净一点"。找到让整个分支、helper、模式、条件判断直接消失的方案。好的代码，回头看是理所当然的。

1

文件不准超过 1000 行

一个 PR 把文件从 1k 行以下推到 1k 行以上？默认不通过。先拆分再提交。

2

不准到处加 if

新增的散乱条件判断、临时特判、随机分支 → 设计问题，不是风格问题。逻辑应该在自己的抽象后面。

3

能跑 ≠ 能合

行为正确但结构变差的代码，不通过。保留行为，重构实现。

4

直接、无聊、可维护

拒绝"魔法"代码。拒绝为了通用性而加的薄抽象层。如果一个 wrapper 不能让代码更清晰，删掉它。

5

类型和边界要干净

不要用 any / unknown / 强制类型转换来掩盖不清晰的数据边界。把边界本身搞清楚。

6

逻辑放对地方，复用已有工具

功能逻辑泄漏到公共路径？用一个 bespoke helper 而不是已有的工具？推回去。

7

不要无谓串行，不要半吊子更新

独立的工作串行执行？相关的更新分两步提交可能只做一半？都是设计味道。

Review 时问什么？ + 自动化审查

Thermo-Nuclear 审查问题

每个有意义的改动，都要过一遍这些问题：

1. 有没有 Code Judo 的机会？能不能用更好的结构让这段代码直接消失？
2. 能不能重新定义问题，让需要的概念、分支、helper 层更少？
3. 这个改动让局部架构变好了还是变差了？
4. 这个抽象真的值得存在吗？还是只是一个 wrapper？
5. 逻辑放对层了吗？有没有把实现细节泄漏到不该去的地方？
6. 有没有重复的条件判断，暗示缺了一个模型或 helper？

Codex Auto-Review

PR（代码合并请求）提交后，CI（持续集成）自动触发 Codex Review。不需要手动操作，等 1-2 分钟查看评论。

自动流程

检测到 PR #58 → 自动触发 Codex Review

Review 完成。发现 2 个问题：1) invite_code 长度硬编码 2) 缺少 null check...
两层审查 = 双保险
Thermo-Nuclear 管代码质量和结构健康。
Codex Review 管逻辑漏洞和实现错误。
一个看骨架，一个看细节。

/e2e-verify — 按验收标准逐条检查

Skill 核心指令（中文翻译）
验证需求文档的验收标准 vs 运行中的代码。只报事实，绝不建议修复。

验证手段优先级（由快到慢）：
1. 数据库查询 — 数据状态验证，置信度最高
2. 命令行 / API 调用 — 行为验证
3. 浏览器自动操作 — 界面渲染和数据显示验证

每条结果：PASS / FAIL / SKIP / BLOCKED。FAIL 一律交 /bugfix。

全 PASS → 提交 PR
FAIL → /bugfix 修复 → 重跑验证（最多 3 轮）
3 轮都 FAIL → 停下来报告人

/bugfix — 7 阶段诊断循环

不是随便改改就交差。像医生问诊一样，系统地排查根因再动手。

Skill 核心指令（中文翻译）
所有 Bug 从这里开始。不允许跳过阶段。

Phase 0 读知识 — 先读已有文档，看有没有已知陷阱
Phase 1 建反馈环 — 建立能快速验证修复的方式（整个方法论的灵魂）
Phase 2 复现 — 确认是那个 Bug，确认可复现
Phase 3 假设 — 列 3-5 个排序假设，追问"为什么发生"而非"发生了什么"
Phase 4 探针 — 每个探针对应一个假设，一次一个变量
Phase 5 修复 — 先写回归测试再修
Phase 6 知识沉淀 — 学到了什么？有什么坑？架构要改吗？

并行开发 — 多个窗口同时干活

一个需求文档写好后，用 /to-issues 把它切成多个独立的小任务（Issue），然后同时开 2-3 个窗口并行干。

水平切片（常见错误）

一个窗口写所有接口（后端），另一个写所有页面（前端）。

问题：合到一起才知道对不对。接口对不上 → 两边都要改 → 互相等。

窗口 A：写所有后端接口
窗口 B：写所有前端页面
→ 合并时才发现对不上

垂直切片（正确做法）

每个窗口做一个完整功能：从数据库到接口到页面，一刀到底。

每个切片独立可验证，不需要等别人。

窗口 A：用户注册（数据库+接口+页面）
窗口 B：用户登录（数据库+接口+页面）
→ 各自独立可测试

/to-issues — 智能切片

AI 读你的需求文档，自动按"垂直切片"原则拆分成多个独立的 Issue（任务工单）。每个 Issue 包含完整的验收标准和依赖关系。

Issue 就是一个写清楚要做什么的任务单，通常发到 GitHub（代码协作平台）上管理。

什么时候能并行？

明显可并行：改界面 + 改后端逻辑 — 界面是用户看到的部分，后端是服务器处理数据的部分，互不干扰。

不确定？就别并行。一个窗口一个窗口来，每个跑完 /e2e-verify 确认没问题再开下一个。
